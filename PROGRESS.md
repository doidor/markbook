## 2026-05-05 — llms.txt to spec, Fluent v9 showcase, story portability planned

**What changed:** Three things in one pass. (1) `dist/llms.txt` now follows the llmstxt.org / Fluent format — H1 + blockquote + intro paragraph + flat bullet list of `[Hierarchical/Path](url): description` links pointing at per-page `dist/llms/<path>.txt` files (which contain the source markdown minus story directives). Pages now also mirror their source-tree path under `dist/` (e.g. `dist/components/Button.html`). (2) Added a global wrapper hook to `@markbook/adapter-react` (`reactAdapter({ wrapper: './preview.tsx' })`) and threaded it through `MarkbookAdapter.wrapperModule` + the entry generator so every story is wrapped in the user's chosen provider before mount. (3) Replaced the toy demo with a Fluent UI React v9 showcase covering Button, Input, Switch, Spinner, Avatar, Card, and Tabs, all auto-wrapped in `FluentProvider` via `examples/react-demo/preview.tsx`. Story portability is planned as v0.5 — see new `ROADMAP.md` and **ADR-0006**.

**Why:** llms.txt was emitting a flat plain-text dump, which contradicted the format the user pointed to. The wrapper hook is what makes "easy on user land" actually work for any real provider-driven library (Fluent, Material, Chakra, theme-ui, …) — without it, every story would have to inline-wrap its provider. Story portability needed an architecture before any code lands; ADR-0006 captures the embed-vs-package split and pins down what becomes user-visible (story IDs, `mount` API).

**Next:** Pagefind (v0.2). Then `markbook dev` (v0.3). Then story IDs need a stable-frontmatter override before v0.5 lands.

## 2026-05-05 — Pixie demo, code blocks under stories, props table, Starlight-style design

**What changed:** Four things, reinforcing each other. (1) Reverted the Fluent UI v9 dep and replaced the demo with **Pixie**, a small in-repo made-up library at `examples/react-demo/src/pixie/` (Avatar, Button, Card, Input, Switch — all richly typed with JSDoc on every prop). (2) Each story now displays its TypeScript source in a `<details>` "Show code" disclosure under the rendered preview; source is extracted via the TS compiler API in new `packages/core/src/code.ts` and Shiki-highlighted at build time. The same code is embedded as a fenced `tsx` block in `llms/<path>.txt`. (3) Added a Storybook-style **props table**, generated by `react-docgen-typescript` in new `packages/core/src/props.ts`. Pages opt in via `component: <relpath>` (and optional `componentExport:`) in frontmatter, and place a `:::props\n:::` directive where the table should render — both HTML and markdown variants are emitted, the latter going into the per-page txt. (4) Reworked the HTML output as a **Starlight-style layout**: sticky top header with brand + (placeholder) search, left sidebar with grouped nav (top-level pages first, then `Components/`), main column with `rehype-slug` heading anchors, and a right-side "On this page" TOC. Refreshed CSS (purple accent, monospace code, sticky panels, responsive breakpoints).

**Why:** Code visibility next to a rendered story is core to docs UX — without it, readers have to leave the page. Props tables are how component libraries communicate their API; doing it from TypeScript types means the table can never lie about the implementation. The Starlight visual rework was the biggest single readability win — grouping nav, padding the content column, and pulling section anchors into a TOC fixes a flat sidebar that didn't scale to the 6-page demo. Reverting Fluent for a fake library was forced by the props-table requirement (`react-docgen-typescript` needs the source files in repo) and is the right move regardless — a lib that ships with Markbook makes the demo dogfood-able forever.

**Next:** Wire Pagefind for actual search (v0.2 — the disabled search input is a placeholder). Then `markbook dev` with HMR (v0.3). Once those land, story-portability prerequisites (stable story IDs via frontmatter `id:`) are the next thing on the road to v0.5.

## 2026-05-05 — Page templates

**What changed:** Added a thin templating layer (new `packages/core/src/template.ts`, threaded through `parse.ts` via a `loadTemplate` callback in `ParseOptions`, wired in `build.ts` to read from `<root>/templates/<name>.md`). Pages opt in with `template: <name>` in frontmatter; the template substitutes `{{ title }}`, `{{ description }}`, `{{ content }}`, and `{{ frontmatter.x.y }}` tokens — substitution only, no conditionals or loops. Created `examples/react-demo/templates/component.md` (H1 + description-as-lede + Props + content slot) and collapsed all 5 Pixie component pages to just `frontmatter` + their story sections — the H1, lede, and props-table boilerplate now lives in one place. `index.md` stays templateless. Added `templatesDir?: string` to `MarkbookConfig` (default `'templates'`) and a `.markbook-content > h1 + p` CSS rule to render the description as a muted lede paragraph. **ADR-0008** captures the design.

**Why:** Five component pages were repeating the same 6-line skeleton (`# Title` / description / `## Props` / `:::props\n:::`). With a real component library that's 30+ pages this becomes a maintenance liability — one structural change means editing every file. Templates fix that with the smallest possible engine: text-substitution only, no logic. KISS wins here because the substituted result is *still markdown* — directives in the template (`:::props`) resolve against the page's frontmatter and behave identically to inline use.

**Next:** Pagefind (v0.2). The template engine could later grow conditionals/loops if real use surfaces a need, but the first thing that probably wants it is "auto-list every story export from a `.stories.tsx`" — which might be cleaner solved at the directive level (`:::stories{src=./Foo.stories.tsx}`) rather than inside templates.

## 2026-05-05 — `templatesDir` accepts an array, demo moves templates out of the default path

**What changed:** Upgraded `MarkbookConfig.templatesDir` from `string` to `string | string[]`. When given an array, the loader searches the directories in order and picks the first `<dir>/<name>.md` that exists, with a clear error listing the searched paths if a template isn't found. Each entry is resolved relative to project root (or accepted as absolute), so templates can live anywhere — sibling repo, monorepo `packages/templates`, etc. To validate, the example demo moved its templates from `templates/` → `_layouts/` and now sets `templatesDir: ['_layouts']`.

**Why:** The user wanted "templates anywhere". Single-string `templatesDir` already covered "any location" via `path.resolve`, but array form unlocks **shared + override** layouts: a project can list a global shared dir first and a per-project override second (or vice versa), and a missing template falls through with a useful error.

**Next:** Pagefind (v0.2). No design follow-up needed — this was a small, clean extension to the v0.1 templates feature.

## 2026-05-05 — v0.2 Pagefind search

**What changed:** Added Pagefind to `@markbook/core` deps and a `runPagefind(outDir)` step in `build.ts` that runs after Vite + llms emission. Uses the Node API (`pagefind.createIndex` → `addDirectory` → `writeFiles`) — no shell-out. The HTML template now (a) links `pagefind-ui.css` from the head, (b) replaces the old disabled search input with `<div id="markbook-search-ui">` in the header, (c) adds a `<script src="pagefind/pagefind-ui.js">` + an inline init that calls `new PagefindUI({ element: '#markbook-search-ui', showSubResults: true, resetStyles: false })`, and (d) marks `<article data-pagefind-body>` so only page content is indexed (nav and TOC are excluded). All paths are relative so subdir pages (`components/Button.html`) correctly point at `../pagefind/`. CSS overrides the `--pagefind-ui-*` variables to map Pagefind's UI onto the existing `--mb-*` theme tokens. **ADR-0009** captures the choices.

**Why:** Search is the single biggest UX gap in a multi-page docs site. Doing it now (rather than later) means every subsequent demo / component-library experiment is searchable for free. Picking Pagefind's bundled UI over `@pagefind/default-ui` saves a dep and one moving part — the UI files come out of `writeFiles` automatically.

**Next:** v0.3 — `markbook dev` with HMR. The current dev story is "edit, `pnpm example:build`, refresh" which doesn't scale beyond toy demos.

## 2026-05-05 — v0.3 `markbook dev` with HMR

**What changed:** Refactored `build.ts` to extract `createContext` (path/plugin resolution) and `writePages` (glob + parse + write entries/HTML to tmpDir) — both `build` and the new `dev` reuse them. Added `dev(config)` that runs `writePages` once with `searchEnabled: false`, then starts Vite's `createServer` against tmpDir with `appType: 'mpa'` and `server.fs.allow: [root]` so it can serve story files outside tmpDir. A separate **chokidar** instance (added as a core dep) watches `docsDir` and every entry of `templatesDir`; on `.md` change/add/unlink it re-runs `writePages` and tells Vite to broadcast `full-reload` over the WS. Story `.tsx` and Pixie component changes hot-reload automatically through `@vitejs/plugin-react`. New CLI command `markbook dev` (with `--port` and `--host` flags), new `dev?: { port?, host? }` field on `MarkbookConfig`, new root `example:dev` script. The HTML generator now takes a `searchEnabled` flag — in dev mode no Pagefind link/script tags are emitted (search is build-time only), and the search slot is omitted from the header so the dev page has no broken UI / 404s. `PagefindUI` init is also defensive (`typeof PagefindUI !== 'undefined'`) for resilience.

**Why:** "edit → `pnpm example:build` → refresh" worked for proving the pipeline but breaks down past a handful of pages and stories. HMR lets authoring feel like working in any modern frontend project — type a prop, see it change. Vite's `server.watcher` doesn't reliably pick up files outside its `root` (which is tmpDir, sibling to docsDir), so a dedicated chokidar is the right architectural split: **Vite watches its module graph (story `.tsx`, components); chokidar watches user content (`.md`, templates).** The two never overlap. The `searchEnabled` flag is small but matters — without it, a dev page would show a broken search input that 404s on Pagefind assets.

**Next:** v0.4 — Vue and web-components adapters. The dev experience is now solid enough to dogfood multi-framework support without authoring friction.
