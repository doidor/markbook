# Markbook roadmap

Forward-looking work, ordered by priority. The current state is **v0.10+** — `:::stories` directive (multi-export story files with CSF v3 support), per-export sliced code disclosures, `@markbook/core` public/internal API split, full READMEs for every package, GitHub Actions CI, **`config.directives` extension model** (ADR-0025) for user-defined directives + the `htmlTemplate(source)` helper for `.html`-templated handler output, **SEO defaults** (canonical + OG + Twitter + sitemap.xml + robots.txt, all emitted in build and dev), a **`public/` folder** for static assets, a **`markbook preview` command** for serving `dist/` locally, the **official Markbook site** (`examples/markbook-site/`) as a self-hosted documentation showcase, **Shiki-highlighted fenced markdown code blocks** with a hover-revealed copy button, and **frontmatter `order:`** for explicit sidebar position.

## Framework adapters — React shipped, Vue + Web Components planned

Today **only the React adapter (`@markbook/adapter-react`) is implemented.** The core engine is deliberately framework-agnostic (it knows nothing about React), so additional adapters are purely additive. The **Vue** and **Web Components** adapters are planned but **not yet implemented** — they were removed from the tree to keep the surface honest until they're built (see ADR-0028). Each will be a thin package mirroring the React adapter's contract: a browser-side `mount()` (reusing `@markbook/adapter-shared`) plus a Node-side config helper, with `:::props` tables, interactive controls, and decorators following once the base mount lands.

## v1.0 — Stable API freeze

Once the deferred carve-outs land (or are explicitly punted to v1.x), freeze `markbook.config.ts`, the three built-in directives (`:::story`, `:::stories`, `:::props`) and their attributes, the **user-directive API** (`config.directives`, `DirectiveHandler`, `DirectiveContext`, `DirectiveResult`, `BUILTIN_DIRECTIVES`, `escapeHtml`/`escapeAttribute`/`htmlTemplate`), the adapter contract, the decorator API, the embed/package output formats, the `--mb-*` theme tokens, the **frontmatter contract** (`title`, `description`, `order`, `template`, `layout`, `ogImage`, `component`, `componentExport`), the customization API (`css` / `disableBaseCss` / `layoutsDir` / `transformHtml`), the SEO output shape (sitemap.xml + robots.txt schema + the canonical/OG/Twitter `<head>` block), the `public/` static-asset contract, the **fenced-code DOM contract** (`.markbook-code-pre-wrap`, `.markbook-fenced-code`, `[data-markbook-copy]`), the CSF v3 detection rule, and the story-file conventions (`args`, `argTypes`, `parameters`, frontmatter `id=`/`component:`/`template:`). After v1.0 these become semver-stable; bump every package to `0.1.0` and ship.

## Deferred (post-freeze, additive)

- **Vue + Web Components adapters.** Rebuild `@markbook/adapter-vue` and `@markbook/adapter-wc` against the existing adapter contract (each consuming `@markbook/adapter-shared` for the placeholder/shadow/CSS plumbing). Includes their `:::props` tables (a `vue-component-meta` path and a custom-elements-manifest path, since `:::props` currently delegates to `react-docgen-typescript` and is React-only), interactive controls, and decorators (WC slot composition is a different mental model — likely a slot-projection helper rather than the React wrapping pattern).
- **Shadow-DOM CSS injection.** Embed bundles inject `<style>` into `document.head`, which doesn't reach shadow roots. The story renders unstyled inside the shadow.
